# ADR-0024: Batch embeddings ŌĆö deferred reindex model

- **Status:** Proposed
- **Date:** 2026-07-10
- **Decision-makers:** phanijapps
- **Supersedes:** none
- **Related:** ADR-0022 (engine grid vs backend recipe), [`docs/specs/atomic-batch-ingest/spec.md`](../specs/atomic-batch-ingest/spec.md) (S3 batch; Evidence/Embeddings `Skipped`), `docs/backlog.md` (`atomic-batch-evidence-embeddings`, resolved by this ADR)

## Decision summary

- **Decision:** The `BatchIngest` **Embeddings** step stays `StepStatus::Skipped` in v1 ŌĆö the **deferred-reindex model**: the batch records `EmbeddingRef` *metadata* via each record's `Provenance`, and the actual vectors are generated by a **separate reindex job** through `engram_retrieval::VectorIndex` (the `SqliteVectorIndex` adapter), not by the batch itself.
- **Because:** wiring embeddings into the batch would force `SqlBatchIngest { memory, knowledge }` to compose a third store handle (the vector index) plus a content-hash-keyed upsert, when a batch is best-effort across two separate SQLite files with no cross-store transaction ŌĆö adding a third store inside the batch does not make it more atomic, only more coupled.
- **Applies to:** `adapters/integration/src/batch.rs` (the Embeddings step reports `Skipped`); `core/integration/src/batch.rs` (the port's step enum keeps the `Embeddings` step).
- **Tradeoff accepted:** a caller must run a separate reindex to make batch-written records retrievable by vector; the batch is not a "write + immediately searchable by embedding" operation.
- **Revisit if:** a deployment wants the batch to own its embedding generation (e.g. a single-store backend where the vector index shares the batch's connection) ŌĆö that triggers an inline-write variant behind the same port.

## Context

S3 ships a best-effort batch across the separate memory and knowledge SQLite
stores. The backlog entry `atomic-batch-evidence-embeddings` deferred the
Evidence and Embeddings steps on one question for embeddings: does the batch own
a `VectorIndex` handle (inline write) or delegate to a follow-up indexing job
(deferred reindex)?

- The `BatchIngest` request already carries `embeddings: Vec<EmbeddingRef>` ŌĆö
  these are **metadata** (id, model, dimensions, target, content_hash), not
  vectors.
- Vector storage lives behind `engram_retrieval::VectorIndex`
  (`SqliteVectorIndex`), a third store the current `SqlBatchIngest` shape does
  not hold.
- The batch is explicitly **not** cross-store ACID (separate SQLite files,
  best-effort, per-step partial-failure reporting); composing a third store
  inside it adds coupling without adding atomicity.

## Decision

Adopt the **deferred-reindex model** for batch embeddings:

> The batch Embeddings step reports `StepStatus::Skipped` ŌĆö honest, not silent.
> The batch records `EmbeddingRef` metadata via the records' `Provenance`; actual
> vectors are generated by a separate reindex job through `VectorIndex`.

Two rules govern it:

1. **`Skipped` is the correct implementation, not a stub.** The step is reported
   `Skipped` (never silently absent); the deferred-reindex model is the decision,
   recorded here, not a "wire it later" TODO.
2. **The port is unchanged.** `BatchStep::Embeddings` stays in the step enum and
   stays last in `ALL_STEPS`; a future inline-write backend reports a different
   status through the same step. No port change, no DTO change.

## Decision drivers

- **Keep the batch two-store and best-effort** ŌĆö adding a vector index inside
  the batch couples three stores for no atomicity gain; the winning driver.
- **Vector generation is a separate concern** ŌĆö embeddings need a query/
  embedding model (feature-gated, off by default), which is naturally a
  background reindex job, not an ingest-time write.
- **Honest reporting** ŌĆö `Skipped` with a decided rationale beats a silent
  no-op or a fake `Succeeded`.

## Consequences

**Positive:**

- `SqlBatchIngest` stays `{ memory, knowledge }` ŌĆö no third store handle, no
  feature-gated embedding model pulled into the batch path.
- Vector generation is decoupled and can run on its own schedule (re-index
  unchanged records only, content-hash dedup) via `VectorIndex`.
- The decision is explicit and referenced from the code, so `Skipped` reads as
  intended, not as unfinished work.

**Negative:**

- A caller that writes a batch and immediately queries by vector gets nothing
  until a reindex runs ŌĆö the batch is not "ingest + vector-searchable".
- Batch-written records' `EmbeddingRef` metadata and the actual vectors can
  drift if the reindex job is not run (the metadata says "an embedding exists"
  before the vector does). Acceptable: the reindex is the source of vector
  truth.

**Revisit if:** a deployment wants the batch to own embedding generation (e.g. a
single-store backend where the vector index shares the batch connection, or an
inline-write requirement) ŌĆö that triggers an inline-write variant reported
through the same `BatchStep::Embeddings` step.

## Confirmation

- **Mode:** existing tests (the batch conformance fixture asserts Embeddings
  `Skipped` on a full batch and re-ingest); this ADR documents that `Skipped` is
  the *decided* behavior, not a deferral. The `adapters/integration/src/batch.rs`
  rustdoc/comment references this ADR at the Embeddings step.
- **Owner:** engram core team.

## Alternatives considered

- **Inline vector write in the batch** (`SqlBatchIngest` composes a
  `VectorIndex`). Rejected for v1: it forces the batch to hold a third store and
  a feature-gated embedding model, coupling ingest to retrieval infra for no
  atomicity gain over the best-effort model. Wins only when a single-store
  backend makes inline write cheap.
- **Drop the Embeddings step from the enum.** Rejected: the step is part of the
  fixed batch order (`ALL_STEPS`) and the DTO (`embeddings: Vec<EmbeddingRef>`);
  removing it loses the honest per-step report and breaks the step-order
  contract. A future backend reports a real status through the same step.
