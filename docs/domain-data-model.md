# Domain Data Model

## Status

V1 core contract accepted; extension contracts draft.

The accepted v1 surface is limited to the entities listed in
`First Contract Slice`. Those shapes are frozen under the compatibility rules in
this document and the versioned artifacts under `contracts/v1/`.

Belief, contradiction, hierarchy, taxonomy evolution, and consolidation models
remain draft extension contracts. They establish direction and naming, but they
are not frozen v1 API commitments until promoted through a later versioned
contract.

## Contract Freeze Policy

Before implementation, this document may change freely.

After a contract surface is marked `Accepted`:

- Do not rename fields.
- Do not remove fields.
- Do not change field meaning.
- Do not change enum value meaning.
- Do not make optional fields required inside the same contract version.
- Do not make nullable fields non-nullable inside the same contract version.
- Do not change identifier semantics.
- Do not make storage-specific fields part of the portable domain model.

Allowed changes after acceptance:

- Add optional fields.
- Add new enum values only when consumers are expected to tolerate unknown
  values.
- Add new entities or request types.
- Add new versioned schemas.
- Deprecate fields without removing them.

Breaking changes require a new contract version.

```text
v1 accepted -> compatible additions only
v2 required -> rename, removal, meaning change, required-field change
```

V1 source of truth:

- Human-readable semantics: this document.
- Machine-readable wire contract: `contracts/v1/schemas/engram-v1.schema.json`.
- Operation examples: `contracts/v1/examples/`.
- Behavior acceptance specs: `specs/v1/`.

Implementation code, generated TypeScript, generated schemas, database schemas,
and API payloads must conform to these artifacts. Rust domain types may become
the generation source later, but they do not override accepted v1 contract files
without an explicit compatibility review.

## Modeling Principles

- Domain contracts are independent of storage engines, APIs, and language
  bindings.
- Rust domain types should be the implementation source of truth after the model
  is accepted.
- Generated JSON Schema and TypeScript types should match the accepted model.
- Every durable record has provenance.
- Every retrieval path enforces policy.
- Memory and knowledge are distinct but composable.
- Source-grounded knowledge is not the same thing as agent memory.
- Deletion, redaction, and retention are first-class domain concerns.
- Unknown future adapters must be able to preserve existing records losslessly.

## Naming Conventions

- Entity names use `PascalCase`.
- Field names use `camelCase`.
- Enum values use `snake_case`.
- Timestamps use RFC 3339 UTC strings at the contract boundary.
- Identifiers are opaque strings. Consumers must not parse them for meaning.
- Metadata maps may exist, but core semantics should use typed fields.

## Shared Value Objects

### Identifier

Opaque stable string identifier.

Rules:

- Must be globally unique within its entity type.
- Should be generated by the engine unless supplied by a trusted caller.
- Must not encode authorization, tenant, timestamps, or storage location in a
  way callers depend on.

Common aliases:

- `MemoryId`
- `EventId`
- `ActorId`
- `SourceId`
- `DocumentId`
- `ChunkId`
- `ConceptId`
- `EntityId`
- `RelationshipId`
- `EvaluationId`

### Timestamp

RFC 3339 UTC timestamp.

Required timestamp meanings:

- `createdAt`: when this domain record was created by the memory layer.
- `updatedAt`: when this domain record was last materially changed.
- `observedAt`: when the source event or fact was observed.
- `validFrom`: when an assertion became true or applicable.
- `validUntil`: when an assertion stopped being true or applicable.
- `expiresAt`: when policy says a record should expire.

### Metadata

Optional string-keyed map for adapter-specific or caller-specific attributes.

Rules:

- Metadata must not be required for core behavior.
- Metadata keys should be namespaced when produced by adapters.
- Metadata values must be JSON-compatible.

Fields:

| Field | Required | Type | Meaning |
|-------|----------|------|---------|
| `tenant` | yes | string | Top-level isolation boundary |

- `tenant` is always required.
- Records from different tenants must never be retrieved together unless an
  explicit federated query type is added later.
- `subject` and `workspace` are policy inputs, not just metadata.

## Belief Network Model

Beliefs are derived stances over evidence. They are not raw memories and should
not be treated as source truth. A belief summarizes what the memory layer
currently thinks is supported by one or more memories, assertions, knowledge
chunks, entities, or concepts.

### Belief

Derived stance about a subject.

Fields:

| Field | Required | Type | Meaning |
|-------|----------|------|---------|
| `id` | yes | BeliefId | Stable belief identifier |
| `scope` | yes | Scope | Belief scope |
| `subject` | yes | BeliefSubject | Subject this belief is about |
| `content` | yes | string | One or more declarative statements |
| `status` | yes | BeliefStatus | Belief lifecycle state |
| `confidence` | yes | number | 0 to 1 confidence |
| `sources` | yes | BeliefSource[] | Evidence supporting this belief |
| `validFrom` | no | Timestamp | Start of truth interval |
| `validUntil` | no | Timestamp | End of truth interval |
| `supersededBy` | no | BeliefId | Replacement belief |
| `stale` | no | boolean | Whether evidence changed and resynthesis is needed |
| `synthesizer` | no | DerivationRef | Synthesis method/version |
| `reasoning` | no | string | Short synthesis rationale |
| `embeddingRefs` | no | EmbeddingRef[] | Embeddings associated with this belief |
| `policy` | yes | Policy | Access and retention policy |
| `provenance` | yes | Provenance | Synthesis provenance |
| `createdAt` | yes | Timestamp | Creation time |
| `updatedAt` | no | Timestamp | Last update |
| `metadata` | no | object | Non-contract attributes |

Rules:

- Beliefs must reference at least one source.
- Beliefs must be recomputable from their sources or explicitly marked as
  manually asserted.
- A single-source belief may be retracted when its only source is invalidated.
- A multi-source belief should be marked `stale` when one source is invalidated
  and resynthesized from the remaining valid sources.

### BeliefSubject

Canonical subject key for a belief.

Fields:

| Field | Required | Type | Meaning |
|-------|----------|------|---------|
| `key` | yes | string | Stable subject key |
| `entityRef` | no | EntityRef | Entity grounding |
| `conceptRef` | no | ConceptRef | Taxonomy grounding |
| `aliases` | no | string[] | Known alternate keys or names |

Rules:

- `key` must be stable within a scope.
- Entity or concept grounding is preferred when available.
- Exact key matching is acceptable for the first slice but should not be the
  only long-term canonicalization strategy.

### BeliefSource

Fields:

| Field | Required | Type | Meaning |
|-------|----------|------|---------|
| `targetType` | yes | enum | `memory`, `assertion`, `event`, `chunk`, `entity`, `relationship`, `concept`, `belief` |
| `targetId` | yes | string | Source object id |
| `weight` | no | number | Contribution weight |
| `confidence` | no | number | Source confidence at synthesis time |
| `validFrom` | no | Timestamp | Source validity start |
| `validUntil` | no | Timestamp | Source validity end |

### BeliefStatus

Enum:

- `active`: eligible for retrieval and composition.
- `stale`: source evidence changed; resynthesis needed.
- `superseded`: replaced by another belief.
- `retracted`: no longer supported by evidence.
- `archived`: retained but not normally retrieved.

### Contradiction

Reviewable disagreement between two or more beliefs, memories, assertions, or

Fields:

| Field | Required | Type | Meaning |
|-------|----------|------|---------|

| `severity` | yes | number | 0 to 1 severity/confidence |

## Knowledge Model

Knowledge records describe source-grounded information external to, or derived
from, corpora such as code repositories, documents, URLs, and uploaded files.
Knowledge graph records describe typed entities and relationships, while
ontologies describe the classes, properties, axioms, and constraints that govern
those graphs. A knowledge graph may be stored in Neo4j, a property graph, RDF,
SQL, or another backend, but the portable model does not encode any backend.

## Hierarchy Model

Hierarchy models multi-level aggregation and navigation. It is related to
taxonomy, but not identical:

- Taxonomy organizes controlled concepts.
- Hierarchy organizes retrievable memory or knowledge objects into abstraction
  layers for navigation and context compression.

Hierarchy nodes may aggregate memories, events, knowledge chunks, entities,
relationships, concepts, or other hierarchy nodes.

### HierarchyNode

Aggregate or base node in a hierarchical memory/knowledge structure.

Fields:

| Field | Required | Type | Meaning |
|-------|----------|------|---------|
| `id` | yes | HierarchyNodeId | Stable hierarchy node identifier |
| `scope` | yes | Scope | Hierarchy scope |
| `kind` | yes | HierarchyNodeKind | Node category |
| `layer` | yes | integer | 0 for base nodes, higher for aggregates |
| `name` | yes | string | Human-readable name |
| `summary` | no | string | Aggregate summary |

| `status` | yes | HierarchyNodeStatus | Lifecycle state |
| `policy` | yes | Policy | Access and retention policy |
| `provenance` | yes | Provenance | Construction provenance |

Rules:

- Layer `0` nodes represent base retrievable objects or direct projections.

  records, not multiple parent pointers in the same tree.

## Invariants

- Every `MemoryRecord` has `scope`, `policy`, and `provenance`.
- Every `KnowledgeSource`, `SourceDocument`, and `KnowledgeChunk` has `policy`
  and `provenance`.
- Retrieval must never return records outside the requester's allowed scope.
- Retrieval must apply policy before context composition.
- Redacted records must not expose redacted content through retrieval,
  evaluation, links, or explanations.
- Forgotten records must not be returned unless an explicit audit API is added.
- `KnowledgeChunk` is source-grounded; it must reference `sourceId` and
  `documentId`.
- `MemoryRecord` may link to `KnowledgeChunk`, but is not a replacement for it.
- `Belief` is derived state and must reference supporting sources.
- Contradictions must be reviewable records, not hidden destructive updates.
- A consolidation cycle that mutates durable state must produce a
  `ConsolidationRun`.
- Hierarchy nodes must preserve enough provenance to explain their source
  members or source algorithm.
- Taxonomy `broader` and `narrower` relations represent direct links only.
- Embedding vectors are not required in portable contract payloads.
- Storage adapters may add indexes and internal tables but must preserve the
  portable domain model.

## V1 Acceptance Decisions

- `Scope.tenant` is required. `subject`, `workspace`, `session`, and
  `environment` are optional policy inputs. V1 implementations must not assume
  `subject` is present.
- `MemoryRecord.status` is required from day one.
- `Provenance.actor` embeds an `Actor`. V1 does not use `actorId` alone because
  provenance must remain portable across stores and bindings.
- `Policy.allowedUses` is optional. When absent, implementations may allow safe
  internal uses except `training_export`.
- `training_export` is excluded from v1 `AllowedUse`. It requires a later
  governance design before appearing in an accepted contract.
- `MemoryAssertion` is deferred from the v1 schema surface. `MemoryRecord`
  keeps an optional `assertions` field in the conceptual model, but accepted v1
  machine contracts do not require assertion read/write behavior.
- `Belief` and contradiction records are deferred from v1 accepted behavior.
  Their draft shapes remain in this document for future promotion.
- Hierarchy nodes are deferred from v1 accepted schemas.
- Consolidation runs are deferred from v1 accepted schemas. Manual write,
  retrieve, and forget operations do not require consolidation records.
- `KnowledgeSource`, `SourceDocument`, and `KnowledgeChunk` are included in v1
  only as source-grounded setup and retrieval targets needed by evaluation and
  future code/document knowledge ingestion. Ingestion behavior is specified
  later.
- `EvidenceRef.quote` is allowed but optional. It must be short, legally safe,
  and never required for correctness when `uri` or `location` exists.
- `metadata` allows arbitrary JSON-compatible values. Core semantics must use
  typed fields, not metadata conventions.

- Which taxonomy evolution operations are safe to automate?
- Should future policy contracts add a governance model for training export?
