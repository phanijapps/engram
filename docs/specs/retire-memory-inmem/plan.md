# Plan: Retire Memory In-Memory Adapter

- **Spec:** [`spec.md`](spec.md)
- **Status:** Done

> at the bottom.

## Approach

Retire `engram-store-memory` by replacing its two active outside uses with
SQLite-backed stores, then deleting the crate and adding a guard that prevents
the broad process-local adapter from returning. The deletion is intentionally
paired with documentation updates because the retired crate used to carry
memory, knowledge, hierarchy, belief, retrieval, and mutating-consolidation
prototype behavior. After retirement, those concerns must be exercised by their
focused durable adapters or future dedicated specs, not by a catch-all test
store.

Files expected to change: `Cargo.toml`, `Cargo.lock`, `core/eval/Cargo.toml`,
`core/eval/tests/fixture_runner.rs`, `adapters/ingest/Cargo.toml`,
`adapters/ingest/tests/*`, `adapters/memory/sqlite/examples/*`,
`adapters/memory/inmem/**`, `.codex/hooks/`, `AGENTS.md`, `README.md`,
`ROADMAP.md`, `docs/architecture.md`, `docs/architecture/overview.md`,
`docs/benchmarks.md`, `examples/README.md`, and `docs/specs/README.md`.

## Constraints

- `engram-memory` remains the source of memory service and repository ports.
- `engram-knowledge`, `engram-belief`, and `engram-hierarchy` stay behind their
  own focused adapters; memory SQL must not absorb them.
- `engram-eval` stays adapter-neutral and only depends on a concrete store in
  dev-dependencies.
- Historical specs remain a ledger. Current docs and validation hooks carry the
  active architecture.

## Design (LLD)

### Design Decisions

- Use `SqlMemoryService::open_in_memory()` for evaluation fixtures. This proves
  the real SQL service path without requiring a file-backed database.
- Use `SqlKnowledgeStore::open_in_memory()` for ingestion tests. Ingestion is a
  knowledge concern and should not depend on memory persistence.
- Add a SQL-backed benchmark example instead of keeping the retired in-memory
  benchmark alive.
- Remove the in-memory consolidation executor with the crate. A future durable
  consolidation executor needs a focused spec because the old implementation
  mixed memory, hierarchy, belief, and consolidation mutations inside one
  process-local adapter.

### Data & Schema

No domain or SQL schema changes are expected. Cargo metadata and lockfile
changes are generated by the workspace dependency change.

### Interfaces & Contracts

No public port or v1 JSON contract changes. This slice changes only concrete
test adapters and examples.

### Component / Module Decomposition

- `engram-store-sql`: memory service conformance and local examples.
- `engram-store-knowledge-sqlite`: ingestion repository test backing store.
- `engram-store-belief-sqlite` / `engram-store-hierarchy-sqlite`: focused
  durable persistence for their own concerns.
- `.codex/hooks`: active-reference retirement guard.

### Failure, Edge Cases & Resilience

- **Fixture regression:** SQL retrieval semantics might not satisfy accepted
  eval fixtures. Mitigation: run `engram-eval` and SQL service tests against the
  same accepted fixtures.
- **Coverage loss:** deleting the old crate removes prototype consolidation
  tests. Mitigation: document that behavior as retired adapter-local prototype
  work, not current durable surface.
- **Hidden dependency:** a manifest or current doc can keep the old package
  alive. Mitigation: retirement hook plus `cargo tree`.
- **Boundary backslide:** a focused SQLite adapter could import a sibling
  adapter. Mitigation: existing forbidden-import hooks stay in validation.
